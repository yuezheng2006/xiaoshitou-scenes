## 🎯 P0 优化（已完成）

### 1. ✅ 新增 QUICK-START.md（Agent 5 秒决策表）

**问题**：SKILL.md 310 行，Agent 需要完整阅读才能开始任务，认知负荷重。

**解决方案**：
- 创建 `QUICK-START.md`（~350 行），提供四种模式决策表
- 每个决策表包含：必读文件（按顺序）+ 可选文件（按需）+ 生成前检查清单 + 生成后检查清单
- 典型场景快查表：用户输入 → 5 秒决策路径

**效果**：
- Agent 无需读完 SKILL.md 全文，按决策表按需加载
- 实物图任务只需读 3 个必读文件，而不是 18 个 references 文件
- SKILL.md 头部增加快速跳转，直接引导到 QUICK-START.md

---

### 2. ✅ 简化 Confirm Gate 并隐藏技术术语

**问题**：
- L1-L4/E1-E2/P1-P7 共 13 项检查，过于复杂
- 用户看到"形象 Confirm Gate: L1 FAIL"会困惑

**解决方案**：
- 简化为 CRITICAL 项：L1-L2（计数+锚点）+ E1（眼睛）+ P1/P3/P6（老杨核心识别）
- 对用户隐藏技术术语，FAIL 时用自然语言说明：
  - ❌ "形象 Confirm Gate: L1 FAIL (extra arm)"
  - ✅ "这张图小石头多了一只手臂，需要重新生成"
- 更新 `common-character-lock.md`，在"交付确认度"章节增加对用户说明示例

**效果**：
- Agent 内部仍做完整检查，但输出用户友好
- 检查项聚焦 CRITICAL，降低 PASS 难度
- 用户体验从"技术黑话"变为"清晰说明"

## 🎯 P1 优化（已完成）

### 3. ✅ 新增 persona-quick-checklist.md（双 IP 快速决策）

**问题**：
- 触发 persona 后需要跳转 4-5 个文件
- `persona-author.md` + `persona-author-identity.md` + `persona-author-assets.md` + `persona-scene-patterns.md` + `common-persona-routing.md`
- 双 IP 图的 QA 流程比单 IP 图长 2-3 倍

**解决方案**：
- 创建 `persona-quick-checklist.md`（~280 行），汇总双 IP 所有 CRITICAL 项
- 六步流程：确认触发（5 秒）→ 选互动场景类型 → 资产组合速查 → 生成前三个 Lock → 生成后 P1-P7 快检 → 双 IP 出图方案推荐格式
- 资产组合速查表：按模式（实物图/手绘图/知识卡/PPT）列出必传/可选资产
- 画面权重参考表：各模式下老杨与小石头的占比

**效果**：
- 触发 persona 后只需读 1 个文件（persona-quick-checklist.md）
- 资产组合决策从"需要跳转阅读 3 个文件"变为"查表 5 秒决策"
- 生成前 Lock 的完整 prompt 片段集中在一处

---

### 4. ✅ 优化返修输出格式（用户友好）

**问题**：
- 返修输出"本轮保留 / 本轮调整 / 返修提示词"过于技术化
- 缺少"为什么会出现这个问题"的解释
- 多轮返修可能重复相同错误

**解决方案**：
- 更新 `common-qa-repair.md`，改为"问题-原因-方案"三段式：
  ```
  🔍 发现的问题：[用自然语言描述]
  💡 为什么会这样：[简短解释原因]
  ✅ 解决方案：[说明将如何调整]
  ```
- 增加"多轮返修追踪"：第 2 次相同问题时换根本性方法
- 技术细节提示词对用户隐藏（除非用户明确要求）

**效果**：
- 返修说明从"技术清单"变为"友好对话"
- 用户理解问题根源，建立信任
- Agent 避免重复相同错误

---

### 5. ✅ 建立单一真相来源（可维护性）

**问题**：
- "2D Flat Lock"出现在 character.md, common-character-lock.md, SKILL.md
- "persona 触发词"出现在 persona-author.md, common-persona-routing.md, SKILL.md
- 修改一处规则需要同步更新 3-5 个文件

**解决方案**：
- 更新 `common-character-lock.md`，去除重复定义，改为引用：
  - "维度锁具体内容：见 `character.md` 的 '2D Flat Lock + Limbs Lock' 章节（唯一定义）"
  - "Persona Lock 具体内容：见 `persona-quick-checklist.md`（快速版）或 `persona-author-identity.md`（完整版）"
- 创建 `ARCHITECTURE.md`（~200 行），记录文档依赖关系：
  - 单一真相来源原则表
  - 文档依赖图
  - Agent 最小读取路径
  - 常见修改场景与影响范围
  - 去重检查清单

**效果**：
- 每个概念只在一处定义，其他文件只引用
- 修改 character.md 后只需检查 QUICK-START.md 的引用是否正确
- 维护者有清晰的依赖图和检查清单

---

### 6. ✅ 新增 mode-decision-matrix.md（模式决策对比）

**问题**：
- 知识卡与 PPT 边界模糊（都支持标题+模块+双 IP+手绘风）
- 用户说"做一张能发朋友圈的方法论图"时，Agent 可能误判
- "一页 PPT"是知识卡还是 PPT 模式？

**解决方案**：
- 创建 `mode-decision-matrix.md`（~200 行）：
  - 一句话区分表
  - 详细对比表（尺寸/方向/张数/信息量/留白/标题/模块/阅读顺序/传播用途）
  - 典型用户输入 → 模式判断映射表
  - 边界案例决策（"一页 PPT" / "方法论图" / "实物图还是手绘图"）
  - 升级模式决策表
  - 快速决策流程图

**效果**：
- 模式路由不确定时，Agent 查表 10 秒内决策
- 边界案例有明确规则：单张优先知识卡，连续多页优先 PPT
- 原则明确："配图"默认正文配图（实物/手绘），"能发/收藏"才用知识卡

## 🎓 使用建议（Agent）

### 首次任务
1. **不要**直接读 SKILL.md 全文
2. **先读** `QUICK-START.md`，按决策表判断模式
3. **按顺序**读取决策表中的必读文件
4. 触发 persona 时，**立即读** `persona-quick-checklist.md`

### 模式不确定时
1. 先查 `mode-decision-matrix.md` 的"典型用户输入"表
2. 边界案例查"边界案例决策"章节
3. 仍不确定时，默认保守：实物/手绘，不主动升级

### 生成后检查
1. **内部**做简化 Confirm Gate（L1-L2 + E1 + P1/P3/P6）
2. **对用户**隐藏技术术语，FAIL 时用自然语言说明
3. 按 `common-qa-repair.md` 的"问题-原因-方案"格式返修
