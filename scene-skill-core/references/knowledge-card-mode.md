# 知识卡模式

## 一句话

竖版、信息更完整的独立传播容器；继承手绘图模式的视觉 DNA（纯白、黑线、克制红橙蓝），但不是"信息量更高的手绘图"——知识卡要能承载标题、模块、步骤、风险、行动、金句、评分等结构，适合被收藏、转发、单独发布。

## 什么时候用

用户说 **知识卡 / 图文号知识卡 / 手机海报 / 方法拆解图 / 收藏图 / 竖版传播图**，或内容本身是方法论、步骤、流程、对比、案例、诊断、课程/章节总览、行动清单，且用户想要"能单独发出去、值得收藏"的效果时使用。

和手绘图模式的分界：

- 手绘图模式：一个认知锚点、一个动作隐喻、留白多、16:9，读完就完了，不追求收藏。
- 知识卡模式：完整内容容器，可以有标题、多个模块、阅读顺序，3:4/4:5 或 9:16，追求"存下来、发出去、回头看"。

如果一篇长文里既有适合手绘图的段落，也有适合知识卡的段落，两种都可以出，不必二选一。

## 尺寸与信息量

读取 `references/common-modes-and-sizes.md`。

知识卡模式默认使用「讲清楚重点」「内容更完整」「完整海报」三档，不用「画面更空」那一档（太空的内容更适合手绘图模式）。默认尺寸 3:4 或 4:5；需要竖版强传播效果时用 9:16 手机海报。

## 结构形态库

知识卡形态只提供方向，不是固定模板；根据内容结构主动匹配或创造形态：

| 内容结构 | 优先形态 | 何时使用 |
| --- | --- | --- |
| 强观点、金句、反常识判断 | 观点卡 | 想让读者先记住一句话 |
| 路径、阶段、增长闭环 | 路线图卡 | 有起点、节点、终点和反馈回流 |
| 方法、框架、模型 | 白板讲解卡 | 当前 profile 主角色适合站旁边讲解 |
| 今天就能做的步骤 | 行动清单卡 | 知识要转成任务、清单、下一步 |
| 课程、章节、模块 | 章节总览卡 | 有课程包装、章节号、学习路径 |
| 评分、诊断、对比评估 | 诊断评分卡 | 有分数、维度、问题、建议 |
| 前后变化、踩坑复盘 | 分镜对比卡 | 有过程、转折、失败和修正 |

如果已有形态不合适，就为当前内容创造新形态，并在推荐原因里说明它如何帮助读者看懂结构；不要硬套参考形态、不要复刻旧知识卡构图。

## 角色分工

默认 profile 触发老杨时，启用**双 IP 互动**（读 `persona-scene-patterns.md`）：

```text
老杨 = 主讲 / 指向 / 批注 / 总结（贴近标题、模块或结论）
小石头 = 执行 Agent（1–3 个）：搬卡、贴标签、标风险、递交反馈、演示失败案例
```

- 方法论 / 步骤 / 流程 / 对比 / 案例 / 风险 / 行动清单类：默认老杨 + 2–3 个小石头分工，不能只有老杨和文字。
- 纯观点卡 / 金句卡：老杨主讲占主视觉，0–1 个小石头执行点缀。
- 每个小石头遵守 `common-character-lock.md` 多人差异锁。
- 未触发 persona 时：一个主角色承担主讲/指向，其余 1–3 个主角色承担步骤里的具体动作（不引入老杨）。
- Persona 规则见 `common-persona-routing.md` 与 profile 下 `persona-author.md`（及 `persona-author-modes.md`）。

## 视觉规范

- 继承 `handdrawn-style-dna.md` 的颜色规则：黑线主体，橙色做主路径/重点，红色做问题/风险，蓝色做补充说明；克制，宁少不多。
- 纯白或近白背景，允许卡片内部用极简分区线/浅色块区分模块，但不要用大面积渐变、炫彩或商业模板色块。
- 必须有清楚的标题、阅读顺序和信息结构——这是和手绘图模式最大的区别，不能只有一个隐喻加几个标签。
- 手写字/中文批注可以比手绘图模式多，但仍要克制字数，宁可拆成两张知识卡，不要把字硬塞进一张。

## 必填字段（结构锁定，生成前必须完成）

```text
模式：知识卡
知识卡形态：
核心观点：
图内标题：
模块/步骤拆分：
角色分工（哪个主角色 / persona 做什么；触发老杨时写老杨 + 小石头执行分工）：
默认尺寸：
图内信息量：
建议中文标注：
必须保留的词：
避免的失败信号：
```

## 硬性预算

- 「讲清楚重点」：1 个主视觉 + 3-5 个短标签，不需要正式标题框。
- 「内容更完整」：1 个标题 + 3-6 个模块/步骤，每个模块 1 个短标签或短句。
- 「完整海报」：1 个标题 + 最多 8 个模块，允许分区，但仍要保证从上到下/从左到右能顺着读完，不能堆成信息墙。
- 无论哪一档，中文正文文字总量都要克制；文字太多先拆成两张知识卡，不要硬塞进一张。

## 失败信号

- 只有一个隐喻和几个标签，没有标题、模块或阅读顺序——图解感不足，判不合格，应回到手绘图模式或补齐结构。
- 方法论/步骤/流程/对比/案例/风险/行动清单类知识卡里只有文字，没有小石头执行分工；触发老杨时还不能只有老杨+文字。
- 硬套参考形态，没有根据当前内容结构设计模块。
- 太像 PPT 页面（见 `ppt-presentation-mode.md` 的区分）：知识卡是独立传播的单张收藏图，不是连续页面的一页。
- 太商业、太精致插画、太幼态可爱。
- 复刻旧知识卡构图，没有为当前内容重新设计结构。
- 主角色维度漂移、多人复制粘贴、persona 识别丢失——同 `physical-qa-checklist.md` / `handdrawn-qa-checklist.md` 已有判定标准。
- 图内文字过多、错字、看不清阅读顺序。

## QA 与返修

先做通用 QA：读取 `common-qa-repair.md` 的装饰性测试、用户反馈映射表和返修输出格式。

知识卡专属检查顺序：

1. 是否像当前 profile 的主角色 / persona 锚点，识别锚点是否保留。
2. 是否有清楚的标题、阅读顺序和信息结构（不是配图感）。
3. 方法论/步骤/流程/对比/案例/风险/行动清单类内容是否安排了多个主角色协作，而不是只有主角+文字。
4. 图内文字是否准确、可读、不过量。
5. 视觉风格是否仍是白底手绘语言（黑线+克制红橙蓝），没有滑向商业信息图或 PPT。
6. 是否硬套参考形态，没有服务当前内容结构。

不合格就重生成或局部编辑，不要把低质量图包装成最终交付。
